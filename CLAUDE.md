# DynaBooks -- Accounting System for DynaNav Systems Inc.

## Project Overview
DynaBooks is a self-hosted double-entry accounting system. It handles
invoicing, journal entries, bank payments, and bank receipts.

## Tech Stack
- Backend: Python Flask + python-accounting library (PyPI)
- Database: SQLite via SQLAlchemy (python-accounting provides the ORM)
- Frontend: React + Tailwind CSS
- PDF: WeasyPrint for invoice and report PDF generation

## Critical Rules
- ALL accounting logic goes through python-accounting. Do NOT write
  custom debit/credit logic. Use the library's transaction types.
- Journal entries MUST balance (debits == credits) before saving.
- Use python-accounting's built-in report generators for all reports.
- Currency is CAD (Canadian Dollar).
- Invoice numbers are auto-generated: INV-YYYY-NNNN format.
- Brand color: navy #1B3A5C. Accent: #2E75B6.

## Code Style
- Python: PEP 8, type hints, docstrings on public functions.
- React: Functional components with hooks. No class components.
- API: RESTful JSON. All endpoints prefixed with /api/.
- Error handling: Return proper HTTP status codes with error messages.

## Testing
- Use pytest for backend tests.
- Every transaction type needs a test verifying correct ledger postings.
- Test that journal entries reject unbalanced entries.

## python-accounting Library Notes
- Import order matters: call `config.configure_database()` BEFORE importing engine.
- Entity must be `session.add()`'d first -- it sets `session.entity` via event listener.
- Currency has `entity_id` (IsolatingMixin) -- create after Entity, then back-link `entity.currency_id`.
- LineItem must be `session.add()` + `session.flush()` before adding to `transaction.line_items`.
- Transaction workflow: create -> add -> flush -> add LineItems -> flush -> add to line_items -> post -> commit.
- Account codes are auto-generated by the library from config base codes.
- Transaction dates must NOT be exactly at reporting period start (raises InvalidTransactionDateError).
